Musarum.

Ecosistema de streaming musical autoalojado con arquitectura cliente-servidor.

TFG · 10/10 2026 Proyecto individual
Java 21 Spring Boot Angular MariaDB Docker Tailscale VPN
— Qué es —

Tu música, tu servidor, tu control.

Musarum nace de una necesidad real: gestionar una audioteca musical privada con la experiencia de un servicio de streaming moderno, sin depender de plataformas de terceros ni perder la propiedad de los archivos.

El resultado es un ecosistema completo que permite subir, organizar, enriquecer y reproducir música en streaming desde cualquier dispositivo a través de una red privada virtual, sin exponer puertos a internet.

— Arquitectura —

Cliente-servidor RESTful sin estado.

El backend expone una API REST consumida por un frontend Angular y una app de escritorio Electron. No hay sesiones en memoria: cada petición lleva un token JWT que se valida criptográficamente antes de llegar al controlador.

Backend

Java 21 + Spring Boot. Arquitectura por capas: Controllers → Services → Repositories. Transacciones con @Transactional, caché con Caffeine, documentación con Swagger.

Frontend

Angular con componentes standalone, lazy loading por ruta, guards de autenticación y rol, interceptor HTTP para inyectar JWT automáticamente.

Persistencia

MariaDB con Spring Data JPA. Índices estratégicos, lazy loading con @EntityGraph para evitar N+1, proyecciones para consultas ligeras.

Infraestructura

Docker Compose orquesta backend + base de datos. Tailscale VPN cifra el tráfico extremo a extremo sin exponer puertos. Tres instancias de respaldo.

— Qué hace —

No es un reproductor. Es un ecosistema.

Streaming con Range Requests

HTTP 206 Partial Content. El navegador solicita fragmentos del fichero, permitiendo seek instantáneo sin descarga completa.

Enriquecimiento automático

Al subir o escanear música, el sistema consulta Deezer, MusicBrainz, iTunes y LRCLib para completar portadas, biografías, géneros, BPM y letras sincronizadas.

Seguridad por roles

Tres niveles jerárquicos (Listener, Contributor, Owner) con JWT + @PreAuthorize. BCrypt para contraseñas. Bloqueo automático tras 4 intentos fallidos.

Analítica personal

Cada reproducción se registra. El usuario consulta sus artistas, géneros y canciones más escuchadas, horas pico y calendario anual — todo calculado en SQL.

Modos de reproducción inteligentes

Además de aleatorio y bucle, modos por tonalidad (rueda de Camelot) y BPM que encadenan pistas como lo haría un DJ.

Escaneo masivo

El Owner escanea un directorio completo del host. Progreso en tiempo real con AtomicInteger. Sin duplicar archivos en disco — solo se indexan rutas.

— Por qué así —

Decisiones técnicas.

¿Por qué stateless?

Sin sesiones en servidor, cada instancia es independiente. Escalar es añadir contenedores, no sincronizar estado.

¿Por qué VPN en vez de HTTPS?

El servidor es doméstico. Exponer puertos a internet invita a ataques. Tailscale cifra todo el tráfico y solo permite acceso a dispositivos autorizados.

¿Por qué no guardar audio en la BD?

Persistir ficheros binarios en MariaDB degrada el rendimiento. Musarum almacena rutas relativas y sirve los archivos desde volúmenes Docker.

¿Por qué Caffeine y no Redis?

Para un servidor de un solo nodo, una caché en proceso es más rápida y sencilla. 15 min de TTL y 500 entradas cubren el caso de uso sin infraestructura extra.

— Diseño —

Identidad mudéjar.

El primer diseño era un clon visual de Spotify. Lo descarté por falta de personalidad. La identidad final se inspiró en la arquitectura andalusí: paleta de tonos oscuros con ornamentación geométrica en oro envejecido, fondo ink, textos en crema y acentos en terracota.

La misma filosofía que aplico en este portfolio — coherencia visual entre lo que construyo y cómo me presento.
